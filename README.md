# 서비스 도메인 핵심 로직 구현 챌린지

## 챌린지 진행 원칙

본 프로젝트는 TDD(Test Driven Development)를 기반으로 진행한다.

테스트를 먼저 작성하며, 현재 레벨에서 요구하는 기능만 구현한다.

---

# 레벨 진행 순서

## 1. 요구사항 확인

현재 레벨에서 구현해야 하는 비즈니스 규칙을 확인한다.

이번 레벨에서 구현할 기능과
구현하지 않을 기능을 명확히 구분한다.

---

## 2. 필요한 객체 설계

이번 레벨에서 필요한 최소 객체를 생성한다.

생성 하위 폴더

```
domain/
service/
exception/
```

객체의 구조(필드, 메서드)만 작성하며,
비즈니스 로직은 구현하지 않는다.

---

## 3. 테스트 시나리오 작성

테스트 클래스를 생성한 후,
구현할 비즈니스 요구사항(테스트 시나리오)을 클래스 상단에 먼저 작성한다.

예시

```
잔액이_충분하면_결제에_성공한다
잔액이_부족하면_결제에_실패한다
결제금액이_0원이면_실패한다
```

---

## 4. 테스트 코드 작성

테스트를 하나씩 작성한다.

Given

When

Then

형태로 작성한다.

---

## 5. 테스트 실패 확인

작성한 테스트가 실패하는 것을 반드시 확인한다.

처음부터 테스트가 통과하면 안 된다.

---

## 6. 요구사항 구현

현재 실패한 테스트를 통과할 만큼만
비즈니스 로직을 구현한다.

미래 레벨 기능은 절대 구현하지 않는다.

---

## 7. 테스트 실행

현재 테스트가 통과하는지 확인한다.

실패한다면 원인을 수정한 후 다시 실행한다.

---

## 8. 다음 테스트 반복

다음 테스트를 추가한다.

```
테스트 작성
    ↓
테스트 실패 확인
    ↓
최소 구현
    ↓
테스트 통과
```

현재 레벨의 모든 테스트가 통과할 때까지 반복한다.

---

## 9. 성능 측정

현재 레벨의 테스트가 모두 성공하면
성능을 측정한다.

측정 항목

- Response Time
- Throughput

필요 시

- 시간복잡도
- 자료구조
- 병목 구간

을 함께 분석한다.

---

## 10. 리팩토링

중복 제거

책임 분리

가독성 개선

성능 개선

을 진행한다.

리팩토링 후에도 모든 테스트가 통과해야 한다.

---

# 구현 원칙

- 구현보다 테스트를 먼저 작성한다.
- 현재 레벨에서 필요한 기능만 구현한다.
- 테스트가 요구사항을 설명해야 한다.
- 실패하는 테스트 없이 구현하지 않는다.
- 미래 레벨 기능을 미리 만들지 않는다.
- 테스트 통과 후 리팩토링한다.
