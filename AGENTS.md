# 서비스 도메인 핵심 로직 구현 챌린지

## 1. 프로젝트 개요

특정 서비스 도메인의 핵심 비즈니스 로직을 TDD 방식으로 직접 구현하고,
처리량·응답 속도·정합성·동시성 등 실제 서비스에 필요한 요구조건을 만족하도록 설계하고 검증하는 학습 프로젝트다.
특정 기술을 많이 사용하는 것이 아니다.
이 프로젝트의 목적은 완성된 서비스를 빠르게 만드는 것이 아니다.

핵심 목적은 다음 역량을 훈련하는 것이다.

- 도메인 분석 능력
- 요구사항 도출 능력
- 테스트 설계 능력
- 핵심 비즈니스 로직 구현 능력
- 예외 상황 대응 능력
- 동시성 및 데이터 정합성 검증 능력
- 성능 측정 및 병목 분석 능력
- 장애 상황 대응 능력
- 서비스가 성장하면서 발생하는 기술 문제를 이해하는 능력

각 도메인에서 실제 서비스가 올바르게 동작하기 위해 반드시 지켜야 하는 비즈니스 규칙과 기술적 문제를 단계적으로 구현하고 검증한다.

---

## 2. 챌린지의 기본 구조

각 도메인 챌린지는 여러 개의 레벨과 미션으로 구성한다.

### 레벨

레벨은 하나의 큰 문제 영역 또는 기능 단위다.

### 미션

미션은 테스트로 검증 가능한 작은 구현 단위다.
모든 기능을 한 번에 구현하지 않는다.
큰 문제를 여러 레벨과 미션으로 나누고, 현재 단계에서 필요한 문제만 해결한다.

---

## 3. 레벨 학습 원칙

해당 레벨 문제를 직접 경험한 뒤 해결 방법을 도입하는 것을 기본으로 한다.

---

## 4. 미션 설계 원칙

각 미션은 테스트로 검증 가능한 단위여야 한다.
미션은 반드시 다음 내용을 명확하게 정의한다.

### 미션 목표

이번 미션에서 검증할 핵심 비즈니스 규칙을 정의한다.

### 미션 구현 범위

이번 미션에 포함되는 기능을 정의한다.

### 제약사항

반드시 지켜야 하는 도메인 규칙을 정의한다.

### 성공/실패 시나리오

---

## 5. 상황 가정 작성 원칙

테스트 시나리오는 추상적인 설명이 아니라 테스트 코드의
Given / When / Then으로 바로 변환할 수 있을 정도로 구체적이어야 한다.

---

## 6. 테스트 우선 원칙

구현 코드보다 해당 기능을 구현하기 위해 필요한 요구 조건을 
먼저 생각한 후 테스트 코드를 작성하면서 설계 구조를 먼저 생각 한뒤 구현을 한다.

---

## 7. 성능 측정 원칙

모든 챌린지에서 K6과 같은 테스트 도구를 사용하여 부하 테스트를 통해 고도화 되는 레벨별로 필요할때 마다
간단하게 응답시간 요청 시간으로 성능 분석을 하는것이 아니라 제대로 테스트 도구를 사용한다.

---

## 8. 프로젝트 문서 참고 규칙

프로젝트의 상세 규칙은 `/docs` 디렉터리에 문서화되어 있다.

AI는 작업을 수행할 때 필요한 경우 아래 문서를 우선적으로 참고한다.

### 참고 대상

| 경로                    | 참고 내용                              |
| --------------------- | ---------------------------------- |
| `/docs/ai-roles`     | ai의 역할  지침             |
| `/docs/challenge/`    |  모든 서비스 도메인에 공통으로 적용되는 챌린지 설계 템플릿 |
| `/docs/challenge/{domain}` | 특정 도메인의 챌린지 진행 기준과 단계별 구현·검증 문서   |
