# [도메인명] 핵심 로직 구현 챌린지 생성 템플릿

## 1. 챌린지 목적

본 챌린지는 [도메인명] 서비스에서 발생하는 핵심 비즈니스 로직과 기술적 문제를 TDD 방식으로 직접 구현하고 검증하는 것을 목표로 한다.
단순 CRUD 구현이나 완성된 서비스 제작이 목적이 아니다.
다음 역량을 단계적으로 학습하는 것을 목표로 한다.

- 도메인 규칙 분석
- 요구사항 도출
- 테스트 시나리오 설계
- 핵심 비즈니스 로직 구현
- 예외 상황 처리
- 데이터 정합성 검증
- 동시성 문제 재현 및 해결
- 성능 측정 및 병목 분석
- 장애 상황 대응

## 3. 챌린지 진행 방식

챌린지는 TDD 방식으로 진행한다.
각 미션은 다음 순서로 진행한다.

1. 미션 요구사항 확인
2. 구현 범위와 제외 범위 확인
3. 테스트 시나리오 작성
4. 테스트 코드 작성
5. 테스트 실패 확인
6. 최소 구현
7. 테스트 통과
8. 예외 상황 검증
9. 성능 측정
10. 결과 분석
11. 다음 미션 진행

모든 기능을 한 번에 구현하지 않는다.

현재 레벨과 미션에서 요구하는 문제만 구현한다.

---

## 4. 챌린지 진행 원칙

- 구현보다 요구사항과 테스트를 먼저 작성한다.
- 테스트는 구현 세부사항보다 비즈니스 결과를 검증한다.
- 현재 레벨의 문제를 직접 경험한 뒤 해결 방법을 도입한다.
- 미래 레벨의 기술이나 해결 방법을 미리 적용하지 않는다.
- 동시성 미션은 성공·실패 건수와 최종 데이터 상태를 검증한다.
- 성능 개선은 성능 분석 도구를 사용해 측정 결과를 근거로 진행한다.

## 7. 테스트 기준

모든 테스트는 비즈니스 결과를 기준으로 작성한다.

### 정상 케이스

- 올바른 요청이 성공하는가
- 기대한 상태로 변경되는가
- 필요한 데이터가 생성되는가

### 예외 케이스

- 잘못된 요청이 실패하는가
- 적절한 예외가 발생하는가
- 잘못된 데이터가 생성되지 않는가

### 실패 시 데이터 유지

- 기존 상태가 유지되는가
- 일부 데이터만 변경되지 않는가
- 트랜잭션이 필요한 경우 전체 작업이 롤백되는가

### 상태 전이

- 허용된 상태 전이가 성공하는가
- 금지된 상태 전이가 실패하는가
- 실패 시 기존 상태가 유지되는가

### 동시성

- 성공 요청 수
- 실패 요청 수
- 최종 데이터 상태
- 최종 데이터 수
- 중복 데이터 존재 여부
- 비정상 상태 발생 여부

## 8. 성능 분석 기준

각 레벨에서는 필요한 시점에 부하 테스트 도구를 사용하여 성능을 측정한다.
기본 측정 지표는 다음과 같다.

- 평균 응답 시간
- P95
- P99
- TPS 또는 RPS
- 성공률
- 실패율

필요한 경우 다음 항목을 추가로 분석한다.

- 시간복잡도
- 자료구조
- DB 쿼리 수
- 인덱스 사용 여부
- 커넥션 풀
- 스레드 풀
- Lock 경합
- Redis 병목
- MQ 적체
- 외부 API 지연

성능 평가는 단순히 빠르거나 느리다고 판단하지 않는다.

동일한 부하 조건에서 구현 전후를 비교하여 개선 효과를 검증한다.
