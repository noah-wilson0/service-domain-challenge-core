# 서비스 도메인 챌린지 미션 작성 양식

## 1. 문서 목적

이 문서는 서비스 도메인 핵심 로직 구현 챌린지의 미션을 일관된 형식으로 작성하기 위한 공통 양식이다.

각 미션은 구현해야 할 범위와 테스트 목록을 명확하게 제시해야 하며, 사용자가 테스트 코드를 작성하기 전에 성공 조건과 실패 조건을 파악할 수 있어야 한다.

---

## 2. 작성 원칙

### 2.1 미션 목표

- 이번 미션에서 검증할 핵심 비즈니스 규칙을 한두 문장으로 작성한다.
- 구현 기술보다 미션을 통해 보장해야 하는 결과를 중심으로 작성한다.

### 2.2 미션 구현 범위

- 이번 미션에서 실제로 구현하거나 검증할 항목만 목록으로 작성한다.
- 제외 범위는 작성하지 않는다.
- 이후 미션에서 다룰 기능이나 기술은 현재 미션의 구현 범위에 포함하지 않는다.

### 2.3 성공 및 실패 시나리오

- 각 시나리오에 번호와 구체적인 제목을 부여한다.
- 시나리오는 테스트 코드의 Given / When / Then으로 변환할 수 있게 작성한다.
- 하나의 시나리오가 여러 규칙을 한꺼번에 검증하지 않도록 분리한다.
- 각 시나리오 아래에 실제로 작성해야 할 테스트를 점 목록으로 명시한다.
- 실패 시나리오에는 예외 발생 여부와 기존 데이터의 유지 여부를 포함한다.

### 2.4 제약사항

- 모든 성공 및 실패 시나리오를 수행한 뒤 미션을 완료했다고 판단하기 위한 최종 조건 한 가지만 작성한다.
- 개별 시나리오에서 이미 설명한 규칙을 반복해서 나열하지 않는다.

---

## 3. 미션 작성 템플릿

아래 양식을 복사하여 각 미션 문서를 작성한다.

```markdown
# 미션 {번호}. {미션 이름}

## 미션 목표

{이번 미션에서 검증할 핵심 비즈니스 규칙과 달성할 결과를 작성한다.}

## 미션 구현 범위

- {구현 또는 검증 대상 1}
- {구현 또는 검증 대상 2}
- {구현 또는 검증 대상 3}

## 성공 시나리오

### 1. {성공 시나리오 제목}

- Given {초기 상태와 준비된 데이터}
- When {사용자가 수행하거나 시스템이 실행할 행동}
- Then {기대하는 결과와 최종 데이터 상태}

검증할 테스트:

- {테스트해야 할 정상 동작 1}
- {테스트해야 할 정상 동작 2}

### 2. {성공 시나리오 제목}

- Given {초기 상태와 준비된 데이터}
- When {실행할 행동}
- Then {기대하는 결과와 최종 데이터 상태}

검증할 테스트:

- {테스트해야 할 정상 동작 1}
- {테스트해야 할 정상 동작 2}

## 실패 시나리오

### 1. {실패 시나리오 제목}

- Given {실패 조건이 만들어진 초기 상태}
- When {실패를 발생시키는 입력 또는 행동}
- Then {발생해야 하는 예외와 기존 데이터의 유지 여부}

검증할 테스트:

- {테스트해야 할 예외 또는 경계 조건 1}
- {테스트해야 할 예외 또는 경계 조건 2}

### 2. {실패 시나리오 제목}

- Given {실패 조건이 만들어진 초기 상태}
- When {실패를 발생시키는 입력 또는 행동}
- Then {발생해야 하는 예외와 최종 데이터 상태}

검증할 테스트:

- {테스트해야 할 예외 또는 경계 조건 1}
- {테스트해야 할 예외 또는 경계 조건 2}

## 제약사항

- {모든 시나리오를 통과한 뒤에도 반드시 만족해야 하는 미션의 최종 완료 조건 한 가지}
```

## 4. 테스트 목록 작성 기준

테스트 목록은 구현 작업을 시작할 때 체크리스트로 사용할 수 있어야 한다.

- 테스트 하나는 하나의 비즈니스 결과를 검증한다.
- `정상적으로 처리된다`와 같은 추상적인 표현을 사용하지 않는다.
- 입력값, 상태값, 중복 기준과 기대 결과를 구체적으로 작성한다.
- `null`, 빈 문자열, 공백 문자열이 서로 다른 경계 조건이면 각각 명시한다.
- 데이터베이스 제약을 검증하는 테스트와 도메인 객체의 생성 규칙을 검증하는 테스트를 구분한다.
- 중복이나 검증 실패가 발생하면 기존 데이터가 변경되지 않는지도 확인한다.

### 나쁜 예

```text
- 예매가 정상적으로 동작한다.
- 잘못된 값을 처리한다.
```

### 좋은 예

```text
- 저장된 예매를 생성된 예매 ID로 다시 조회할 수 있다.
- 사용자 ID가 공백으로만 구성되어 있으면 예매를 생성할 수 없다.
- 동일한 운행 좌석에는 예매를 중복 저장할 수 없다.
```

## 5. 미션 작성 완료 점검표

- 미션 목표가 비즈니스 결과를 설명하는가?
- 구현 범위에 이번 미션에서 다룰 항목만 포함되어 있는가?
- 성공 시나리오와 실패 시나리오에 각각 번호가 있는가?
- 모든 시나리오가 Given / When / Then으로 작성되어 있는가?
- 각 시나리오 아래에 작성해야 할 테스트 목록이 있는가?
- 실패 시 예외와 기존 데이터의 유지 여부가 명시되어 있는가?
- 제약사항이 미션의 최종 완료 조건 한 가지로 작성되어 있는가?
